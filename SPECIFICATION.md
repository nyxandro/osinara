# Семейный персональный AI-агент на Eve

<!-- markdownlint-disable MD013 MD029 -->

## 1. Статус документа

- Статус: первоначальная спецификация.
- Дата фиксации: 11 июля 2026 года.
- Базовый фреймворк: [Vercel Eve](https://github.com/vercel/eve).
- Проверенная версия Eve на момент подготовки: `0.22.5`.
- Назначение: зафиксировать согласованные продуктовые, архитектурные и безопасностные требования до начала реализации.

Eve находится в активной pre-1.0 разработке. При реализации версия должна быть закреплена точной записью в `package.json` и lock-файле. Обновление Eve допускается только после прохождения тестов и проверки миграционных замечаний.

## 2. Видение продукта

Необходимо создать единого персонального AI-агента, доступного через Telegram и в дальнейшем через другие каналы. Агент должен поддерживать владельца и приглашенных членов семьи, сохранять долговременную память, работать в личных чатах, закрытых семейных группах и изолированных гостевых или рабочих группах, выполнять действия во внешних сервисах и отправлять проактивные уведомления.

Разные каналы являются интерфейсами одного центрального агента. Память, права доступа, задачи и интеграции не должны дублироваться для каждого канала.

## 3. Цели

Система должна:

- предоставлять владельцу личного агента через Telegram;
- позволять владельцу безопасно приглашать членов семьи;
- поддерживать отдельные личные диалоги каждого участника;
- участвовать в зарегистрированных семейных, гостевых и рабочих Telegram-группах;
- разделять личную, семейную, групповую и служебную память;
- поддерживать долговечные сессии, переживающие рестарты и развертывания;
- выполнять чтение и изменение данных во внешних сервисах с необходимыми разрешениями;
- запрашивать подтверждение перед чувствительными или необратимыми действиями;
- создавать личные и семейные напоминания;
- вести диагностируемый журнал действий без раскрытия содержания личных разговоров;
- в дальнейшем подключать веб-интерфейс и дополнительные каналы без создания отдельного агента.

## 4. Границы первой реализации

В первую производственную вертикаль входят:

- один владелец и одна семейная область;
- приглашение нескольких членов семьи;
- личные Telegram-чаты;
- одна или несколько зарегистрированных Telegram-групп с явно заданным типом и областью памяти;
- долговременная память в PostgreSQL;
- гибридный поиск по памяти через PostgreSQL full-text search и `pgvector`;
- управление памятью: создание, просмотр, исправление, удаление и экспорт;
- личные и групповые напоминания;
- одна первая внешняя интеграция;
- подтверждение действий через Telegram;
- аудит, наблюдаемость, резервное копирование и восстановление.

Веб-интерфейс и дополнительные каналы предусматриваются архитектурой, но не входят в первую производственную вертикаль без отдельного согласования.

## 5. Термины

| Термин | Определение |

| Владелец | Пользователь, создавший семейную область и управляющий участниками, группами и общими интеграциями. |
| Член семьи | Приглашенный и подтвержденный Telegram-пользователь. |
| Семейная область | Изолированный контур пользователей, общей памяти, групп, задач и интеграций. |
| Личная память | Данные, доступные только конкретному пользователю в его личном контексте. |
| Семейная память | Данные, доступные подтвержденным членам одной семейной области. |
| Семейная группа | Закрытая зарегистрированная группа семьи, использующая общую семейную память. |
| Внешняя группа | Зарегистрированная гостевая, дружеская или рабочая группа с изолированной групповой памятью. |
| Групповая память | Изолированный контекст внешней Telegram-группы или ее темы, не имеющий доступа к памяти семьи. |
| Служебная память | Настройки, членство, роли, разрешения и технические метаданные. |
| Долговечная сессия | Сессия Eve, сохраняющая историю и состояние между запросами и рестартами. |

Eve отвечает за агентный цикл, долговечные сессии, каналы, инструменты, подключения, расписания, sandbox и human-in-the-loop. Прикладной слой отвечает за пользователей, семейные области, долговременную память, права, аудит и бизнес-правила.

## 7. Пользователи и роли

### 7.1. Обязательные роли

- `owner`: управляет семейной областью, участниками, зарегистрированными группами и общими интеграциями.
- `recovery_owner`: заранее подтвержденный резервный владелец, который может восстановить управление при утрате доступа основным владельцем, но не получает обычные административные права до активации восстановления.
- `member`: пользуется личным агентом и разрешенными семейными возможностями.

### 7.2. Основные правила доступа

- Владелец не получает автоматический доступ к содержанию личных разговоров и личной памяти других участников.
- Резервный владелец не получает доступ к личным данным или административным функциям только на основании резервного статуса.
- Член семьи не получает доступ к личной памяти другого участника.
- В закрытой семейной группе агент использует общую семейную память.
- Индивидуальная личная память участников не используется в закрытой семейной группе.
- В закрытой семейной группе запустить обработку сообщения могут только подтвержденные члены семьи; обращения остальных участников игнорируются до запуска модели.
- Во внешней группе агент использует только память этой группы и не извлекает личную или семейную память.
- Семейная память доступна только подтвержденным участникам соответствующей семейной области.
- Управление участниками, группами и общими подключениями доступно только владельцу.
- Идентификаторы пользователя и семейной области всегда выводятся из проверенного контекста сессии, а не принимаются от модели или из параметров инструмента.

## 8. Регистрация владельца и приглашение семьи

### 8.1. Владелец

Первый владелец создается через одноразовый bootstrap-код в личном Telegram-чате.

Процедура:

1. Bootstrap доступен только пока в БД не существует ни одной семейной области и владельца.
2. Оператор генерирует криптографически случайный код отдельной командой внутри runtime-контейнера.
3. В БД сохраняется только хэш кода, время создания, срок действия и счетчик неуспешных проверок.
4. Код действует `15` минут и не выводится в логи или интерфейсы наблюдаемости.
5. Первый владелец отправляет код боту в приватном bootstrap-диалоге обычным сообщением.
6. Система проверяет Telegram webhook, тип приватного чата, код и отсутствие существующего владельца.
7. После успеха код немедленно инвалидируется, создаются пользователь, семья и membership `owner`.
8. После появления владельца bootstrap отключается без возможности повторного включения обычным Telegram-сообщением.

После `5` неуспешных попыток код блокируется и оператор должен сгенерировать новый. Ответ при ошибке не должен раскрывать, существовал ли похожий код и какая часть проверки завершилась неуспешно.

### 8.2. Приглашение участника

Согласованный пользовательский поток:

1. Владелец создает одноразовое приглашение.
2. Приглашаемый пользователь открывает ссылку или команду запуска Telegram-бота.
3. Система получает Telegram user ID из проверенного webhook-события.
4. Владелец подтверждает добавление пользователя.
5. После подтверждения создается членство пользователя в семейной области.
6. Использованное приглашение становится недействительным.

Приглашение должно иметь ограниченный срок жизни, быть одноразовым и храниться в виде, не позволяющем использовать утечку базы как готовое приглашение. Конкретная продолжительность должна быть задана именованной константой после согласования политики.

### 8.3. Отзыв доступа

Владелец может удалить участника. После удаления новые запросы пользователя должны отклоняться, его активные разрешения и сессии должны быть отозваны в соответствии с политикой удаления данных. Личная память удаленного участника не должна автоматически становиться семейной.

Взрослый участник может самостоятельно покинуть семью. Перед выходом агент предлагает экспорт личной памяти, файлов, календарных метаданных и списка подключений. После подтверждения выхода личные сессии, память, embeddings, файлы, workspace, подключения и связанные индексы удаляются. Созданные участником семейные записи остаются семейными, а автор отображается как бывший участник без сохранения лишних персональных данных.

### 8.4. Восстановление владельца

Каждая семейная область должна иметь одного основного и одного заранее подтвержденного резервного владельца. Резервный владелец должен принять назначение со своего проверенного Telegram-аккаунта до того, как механизм восстановления станет активным.

После принятия назначения резервному владельцу один раз показывается криптографически случайный recovery code. В БД хранится только хэш. Код необходим для запуска восстановления, не передается модели и не выводится в логах. При утрате кода основной владелец может отозвать старый и выпустить новый; резервный владелец должен подтвердить получение нового кода.

Система должна иметь процедуру восстановления управления при потере основным владельцем Telegram-аккаунта. Восстановление не может основываться только на сообщении с нового Telegram-аккаунта или знании публичных семейных данных.

До выбора конкретного механизма обязательны следующие инварианты:

- восстановление выполняется вне обычного агентного диалога;
- операция инициируется подтвержденным резервным владельцем после проверки recovery code;
- основной владелец получает уведомление по всем доступным зарегистрированным каналам и может отменить процедуру до ее завершения;
- все действующие сессии прежнего Telegram-аккаунта отзываются;
- смена владельца фиксируется в аудите;
- семейная память и подключения не передаются новому аккаунту без успешной проверки фактора восстановления.

Защитная задержка составляет `48` часов. Резервный владелец инициирует восстановление, после чего основной владелец получает уведомления по всем доступным каналам и может отменить процедуру. По завершении задержки резервный владелец должен повторно подтвердить активацию. Резервный владелец может быть заменен только основным владельцем, а замена требует подтверждения нового резервного владельца.

## 9. Поведение Telegram-бота

### 9.1. Личные чаты

- Каждый подтвержденный участник имеет отдельный личный контекст.
- В личном чате доступны личная и семейная память пользователя.
- Неподтвержденный пользователь получает безопасный отказ или поток приглашения, но не доступ к агенту.
- Вложения разрешаются только по явной политике типов и максимального размера.

### 9.2. Семейные группы

Бот работает только в группах, зарегистрированных владельцем. Добавление бота в неизвестную группу не дает участникам доступ к памяти или инструментам.

При регистрации владелец обязательно задает тип группы:

- `family_private`: закрытая группа семьи, использующая общую семейную память;
- `external_private`: закрытая гостевая, дружеская или рабочая группа с собственной изолированной памятью;
- `external_public`: открытая группа с собственной изолированной памятью и отдельной политикой инструментов.

Тип группы является частью авторизационного контекста и не определяется моделью. Изменение типа требует подтверждения владельца и не переносит память между областями автоматически.

По умолчанию Eve запускает обработку группового сообщения только в следующих случаях:

- команда адресована боту;
- присутствует упоминание `@bot_username`;
- пользователь отвечает на сообщение бота.

Неприглашенные сообщения группы не должны запускать модель. Изменение этого поведения требует отдельного решения из-за приватности, стоимости и риска фоновой обработки обычных разговоров.

В `family_private` доступна только общая семейная память; индивидуальная личная память участников в групповой контекст не загружается. В `external_private` и `external_public` недоступны личная и семейная память, семейные подключения, семейные расписания и другие семейные данные. Такие группы используют собственную групповую память и индивидуальный allowlist инструментов, заданный владельцем при регистрации группы.

В `family_private` сообщения от Telegram-аккаунтов без активного членства в семье игнорируются до вызова модели, даже если аккаунт находится в группе. Внешние группы допускают обращения участников согласно Telegram-доступу группы и своей политике команд.

Инструменты, память и настройки внешней группы изменяет только владелец агента. Статус Telegram admin сам по себе не дает административных прав в агенте.

При удалении внешней группы из агента владелец подтверждает необратимую операцию. После подтверждения удаляются групповая память, embeddings, сессии, workspace, файлы, расписания и настройки этой группы. Архив по умолчанию не создается.

### 9.3. Темы и цепочки

- Telegram forum topics должны сохранять отдельный контекст через `message_thread_id`.
- Групповые сессии Eve привязываются к сообщению бота, поэтому разные цепочки ответов могут продолжать разные сессии.
- Проактивное сообщение в группу должно указывать зарегистрированный `chatId` и, при необходимости, `messageThreadId`.

### 9.4. Представление ответов

- Ответы длиннее ограничения Telegram должны безопасно разбиваться на несколько сообщений.
- Базовый Telegram-канал Eve отправляет plain text; форматирование должно добавляться только через отдельный безопасный обработчик.
- Непроверенный текст модели и пользователя нельзя напрямую интерпретировать как небезопасную разметку.

### 9.5. Вложения

Первая версия поддерживает изображения, PDF, основные Office-документы и голосовые сообщения. Hosted Telegram Bot API ограничивает скачивание входящего файла `20 МБ`; постоянный workspace и исходящая отправка документа поддерживают до `50 МБ`, а Telegram-фотография до `10 МБ`. Поддержка вложений реализуется через явную политику для каждого типа группы и личного чата.

Обязательные меры:

- allowlist разрешенных MIME-типов и расширений;
- именованные ограничения размера файла и количества файлов в одном сообщении;
- проверка фактического типа файла, а не только заявленного MIME-типа;
- антивирусная проверка до разбора содержимого;
- изолированный разбор документов;
- сохранение исходных bytes в той же trust zone без использования EXIF и иных метаданных для авторизации или маршрутизации;
- отдельное согласие на сохранение извлеченных данных в память;
- хранение оригинала и производных данных до явного удаления пользователем;
- полное удаление оригинала, embeddings и производных артефактов по запросу;
- запрет передачи содержимого неподтвержденному инструменту или провайдеру.

Изображения передаются выбранной vision-модели после проверки типа и безопасности. PDF и Office-документы разбираются в изолированной среде по их нативной структуре либо представляются vision-модели как изображения страниц. Голосовые сообщения проходят транскрибацию через явно выбранного провайдера. Извлеченный текст наследует область доступа исходного чата и не становится долговременной памятью без прохождения обычной политики автоматического сохранения.

Точный allowlist расширений, vision-маршрут и speech-to-text провайдер фиксируются отдельной политикой вложений. Оригинал, извлеченный текст и производные данные хранятся до явного удаления пользователем.

Изображения и визуальные страницы документов анализируются разрешённой владельцем vision-моделью через model registry. Содержимое передаётся только выбранному внешнему модельному маршруту с соблюдением области доступа исходного чата.

Голосовые сообщения транскрибируются через настроенный model proxy или API speech-to-text provider. Локальный Whisper не входит в первую конфигурацию CPU-only сервера. Транскрипция получает ту же область доступа и срок хранения, что и исходное голосовое сообщение.

### 9.6. Стиль общения и обратная связь

Базовый характер агента теплый, дружелюбный и адаптивный. Агент может учитывать сохраненные предпочтения конкретного участника, не меняя общие правила безопасности и доступа.

Правила ответа:

- по умолчанию отвечать кратко и переходить к подробному объяснению по просьбе или когда без него нельзя безопасно выполнить задачу;
- на голосовое сообщение по умолчанию отвечать текстом;
- отправлять голосовой ответ только по явной просьбе пользователя;
- при продолжительной задаче сообщать о начале, значимых этапах и завершении короткими статусами;
- не показывать пользователю внутренний reasoning модели;
- не перегружать чат низкоуровневыми tool calls, если пользователь не запросил технический журнал;
- при ошибке сообщать понятную причину, доступное действие пользователя и стабильный код ошибки.

### 9.7. Управление через Telegram

В первой версии все пользовательские и административные настройки выполняются в Telegram. Отдельная web-панель не требуется.

Пользователь взаимодействует с агентом обычным языком. Для управления семьей, группами, памятью, моделями, бюджетом, подключениями и расписаниями не требуется запоминать Telegram-команды. Агент распознает намерение, показывает текущее состояние, задает недостающие вопросы и использует inline-кнопки для выбора или подтверждения.

Критические операции нельзя выполнять только на основании неоднозначной фразы. Перед удалением, изменением прав, переключением model provider, подключением аккаунта или внешним действием агент показывает точное действие и запрашивает структурированное подтверждение.

Служебные Telegram-команды могут существовать только для обязательного Bot API onboarding, но не являются единственным способом выполнить пользовательскую операцию.

## 10. Модель памяти

### 10.1. Области памяти

| Область | Где используется | Кто имеет доступ |
| --- | --- | --- |
| Личная | Личный чат пользователя | Только соответствующий пользователь |
| Семейная | Личные чаты и группы `family_private` | Подтвержденные члены семьи |
| Групповая | Группа `external_private` или `external_public` и ее темы | Участники группы согласно ее политике |
| Служебная | Авторизация и работа системы | Только прикладной код и разрешенные административные операции |

### 10.2. Типы долговременной памяти

- профиль: имя, язык, часовой пояс и другие явно подтвержденные сведения;
- предпочтения: устойчивые пожелания к поведению и формату ответов;
- факты: проекты, отношения, объекты и иной полезный долгосрочный контекст;
- эпизоды: краткие итоги значимых разговоров и решений;
- общие семейные сведения: планы, списки, правила и договоренности, явно сохраненные в семейной области.

### 10.3. Атрибуты записи памяти

Каждая запись должна содержать как минимум:

- стабильный идентификатор;
- обязательный идентификатор семейной области;
- область доступа;
- идентификатор владельца, группы или темы согласно области;
- тип памяти;
- содержимое;
- источник и ссылку на исходное событие, когда применимо;
- время создания и изменения;
- уровень уверенности или признак пользовательского подтверждения;
- признак чувствительности;
- статус удаления или срок хранения, если политика его предусматривает.

### 10.4. Запись памяти

Агент должен поддерживать явные команды:

- «Запомни, что...»;
- «Сохрани это для семьи»;
- «Исправь воспоминание...»;
- «Забудь...».

Агент автоматически сохраняет подходящие устойчивые факты и предпочтения, после чего обязан уведомить пользователя о содержании и области новой записи.

Автоматическая запись подчиняется следующим правилам:

- личный чат создает личную память соответствующего пользователя;
- `family_private` создает общую семейную память;
- `external_private` и `external_public` создают только память соответствующей группы;
- любой участник внешней группы, обращение которого принято ботом, может создать автоматическую память только в области этой группы;
- групповая запись хранит Telegram-автора исходного сообщения;
- модель не может изменить область памяти, выведенную из проверенного контекста канала;
- уведомление содержит краткое сохраненное значение, область памяти и действие для немедленного удаления;
- пользователь может исправить или удалить автоматически созданную запись;
- секреты и запрещенные категории из раздела 10.6 не сохраняются;
- одноразовые запросы, неподтвержденные предположения модели и быстро устаревающие данные не считаются долговременной памятью;
- при недостаточной уверенности агент не сохраняет запись и задает уточняющий вопрос.

### 10.5. Чтение памяти

- Перед ходом извлекаются только релевантные записи из разрешенных областей.
- Ограничение семейной области и пользователя является частью запроса к хранилищу, а не постфильтром после поиска.
- Поиск является гибридным и объединяет структурированные фильтры, PostgreSQL full-text search и similarity search через `pgvector`.
- Результаты полнотекстового и векторного поиска объединяются детерминированным ранжированием с учетом релевантности, подтвержденности и времени обновления.
- Сохраненный текст рассматривается как недоверенные пользовательские данные, а не как системные инструкции.
- Во внешней группе личная и семейная память никогда не добавляются в системный контекст.
- Объем подставляемой памяти должен быть ограничен политикой контекста и токенов.

Каждый embedding chunk хранит идентификатор и версию embedding-модели. Смена модели выполняется через контролируемую переиндексацию без смешивания несовместимых векторов в одном поисковом пространстве.

Embeddings строятся локальной закреплённой `intfloat/multilingual-e5-small` на CPU. Длинные записи разбиваются на bounded source-aligned chunks без скрытого усечения, а query/passages используют разные E5-префиксы. Содержимое памяти не отправляется внешнему embedding API. Embedding worker запускается как отдельный локальный сервис или процесс с ограниченными ресурсами.

### 10.6. Запрещенные данные

Запрещено сохранять в долговременную память:

- пароли;
- токены и API-ключи;
- приватные ключи;
- одноразовые коды;
- полные платежные реквизиты;
- иные секреты, для которых должно использоваться специализированное защищенное хранилище.

### 10.7. Управление данными

Пользователь должен иметь возможность:

- просмотреть свою память;
- исправить запись;
- удалить запись;
- экспортировать личные данные;
- запросить удаление своих данных в соответствии с согласованной политикой.

Общую семейную запись могут исправить или удалить ее автор и владелец семьи. Остальные участники могут сообщить об ошибке, но не изменяют запись напрямую. Владелец не получает через это правило права изменять личную память участника.

При подтвержденном запросе полного удаления пользователя его личные сессии, память, embeddings, файлы, workspace, подключения и индексы удаляются сразу из рабочего хранилища. Операция необратима после завершения. Копии в уже созданных резервных архивах не восстанавливаются обратно в рабочую систему и исчезают по мере обычного удаления соответствующих backup.

Долговременная память хранится бессрочно до явного исправления, удаления пользователем или удаления соответствующей семейной области. Автоматический TTL для записей памяти не применяется. Удаление записи должно удалить ее из основной таблицы, полнотекстового индекса, `pgvector`, кэшей и последующих резервных копий согласно политике истечения backup.

`defineState` Eve нельзя использовать как замену долговременной памяти: это состояние конкретной сессии, а память должна быть доступна новым сессиям и каналам.

## 11. Сессии и межканальная работа

- Личный Telegram-чат каждого пользователя имеет управляемую последовательность долговечных сессий.
- Групповые цепочки и темы имеют отдельные контексты.
- История Telegram и будущего веб-интерфейса не обязана быть одной физической сессией Eve.
- Межканальная непрерывность обеспечивается общей авторизацией, долговременной памятью, задачами и интеграциями.
- Сообщения в одну активную сессию должны передаваться последовательно; прикладной слой не должен полагаться на Eve как на долговечную FIFO-очередь конкурентных сообщений.
- При высокой частоте сообщений канал должен иметь собственную очередь и передавать следующий ход после перехода сессии в состояние ожидания.

### 11.1. Политика контекста и ротации

Для первой реализации фиксируются следующие параметры:

- compaction запускается при заполнении `75%` контекстного окна модели;
- активная сессия ротируется после `30` календарных дней бездействия;
- активная сессия ротируется после `250` завершенных пользовательских ходов независимо от срока;
- сессия не ротируется во время активного tool call, незавершенного подтверждения, OAuth-потока или другой припаркованной операции;
- после завершения такой операции ротация выполняется перед следующим обычным пользовательским ходом;
- пользователь может немедленно начать новый контекст обычной просьбой, например «начни новый разговор»;
- compacted summary остается частью соответствующей долговечной сессии Eve и не становится долговременной памятью автоматически;
- факты, необходимые между сессиями, сохраняются только через контролируемый memory store;
- reasoning events не сохраняются в пользовательской истории и не показываются в Telegram;
- завершенные сессии хранятся `90` дней, после чего удаляются вместе с compacted summaries, если их сохранение не требуется для незавершенного юридического или безопасностного инцидента;
- технические audit events не содержат текст диалога и подчиняются отдельной политике хранения.

Порог в `75%` оставляет запас для инструментов и результата compaction. Ротация по бездействию и количеству ходов ограничивает разрастание одной Telegram-сессии, а долговременная память сохраняет полезный контекст между сессиями.

## 12. Инструменты и внешние интеграции

Поддерживаемые способы подключения:

- типизированные TypeScript tools;
- MCP connections;
- OpenAPI connections;
- пользовательский OAuth;
- общие сервисные подключения агента.

### 12.1. Личные подключения

Личный календарь, почта и другие пользовательские сервисы подключаются к стабильной идентичности конкретного участника. Токены не должны попадать в модель, историю сессии, память или sandbox.

### 12.2. Семейные подключения

Общий календарь и другие семейные сервисы принадлежат семейной области. Создавать, изменять и отзывать такие подключения может владелец.

### 12.3. Принцип минимальных прав

Каждое подключение получает только необходимые OAuth scopes и разрешения. Отсутствующий обязательный токен или конфигурация приводит к явной диагностируемой ошибке; система не должна продолжать работу с придуманным или резервным значением.

Первой внешней интеграцией является Google Calendar.

### 12.4. Eve default harness

Подтвержденным членам семьи разрешается использовать возможности Eve default harness: web, работу с файлами, sandbox shell и делегирование subagents. Разрешение относится к семейному контуру и не означает доступ к секретам app runtime.

Обязательные ограничения:

- shell и файловые операции выполняются только в изолированном sandbox;
- sandbox не получает переменные окружения и токены подключений;
- сетевой egress sandbox задается явной политикой;
- доступ к семейным и личным внешним сервисам выполняется через типизированные tools или connections;
- опасные внешние действия по-прежнему требуют подтверждения;
- максимальная глубина, количество subagents и токен-бюджет задаются в конфигурации агента;
- каждая внешняя группа получает собственный allowlist application, integration, shell, network и orchestration инструментов, который владелец задает и изменяет явно;
- отсутствие такого инструмента в allowlist внешней группы означает запрет; нативные file-only операции внутри изолированного group workspace являются базовой возможностью.

Если инструмент внешней группы создает внешний побочный эффект, участник группы может только подготовить действие. Выполнение подтверждает основной владелец агента независимо от Telegram-роли инициатора.

### 12.5. Google Calendar

Первая интеграция должна поддерживать личные календари участников и общий семейный календарь. Личные OAuth grants принадлежат конкретным пользователям, а подключение общего календаря принадлежит семейной области и управляется владельцем.

Изменять общий семейный календарь могут все члены семьи после подтверждения конкретного действия.

Минимальные возможности:

- просмотр доступных календарей и событий;
- получение свободных интервалов;
- создание события после требуемого подтверждения;
- изменение времени, состава участников и описания после подтверждения;
- удаление или отмена события после подтверждения;
- добавление личных и семейных напоминаний на основании календарных событий;
- явное отображение календаря, аккаунта, времени, часового пояса и участников до подтверждения записи.

Модель не выбирает Google-аккаунт по текстовому идентификатору пользователя. Аккаунт определяется проверенной идентичностью сессии и выбранным пользователем календарем. Требуемые OAuth scopes будут зафиксированы отдельной матрицей разрешений.

Каждый участник подключает собственный Google-аккаунт по OAuth-ссылке, отправленной только в его личный Telegram-чат. OAuth callback связывает grant с проверенной Telegram-идентичностью, которая инициировала подключение. Владелец отдельно выбирает из доступных ему календарей общий семейный календарь.

### 12.6. Web-сессии и credential broker

Агент может входить в разрешенные web-сервисы через защищенный credential broker. Модель, sandbox и содержимое сессии не получают логины, пароли, cookies, recovery codes или 2FA secrets.

Обязательные правила:

- credential broker связывает авторизацию с конкретным пользователем и доменом;
- пользователь явно подключает web-аккаунт до первого использования;
- токены и cookies инъектируются на доверенной границе и не возвращаются модели;
- подключение имеет allowlist доменов и может быть отозвано пользователем;
- интерактивный 2FA или подтверждение входа выполняет сам пользователь;
- web-сессия одного участника недоступна другому участнику, семейной или внешней группе;
- ошибка broker не приводит к запросу пароля в обычном Telegram-сообщении.

### 12.7. Покупки и платежи

Агент может искать товары, сравнивать условия, собирать корзину и подготавливать заказ. Финальное оформление покупки, отправка платежного поручения или иное действие, создающее финансовое обязательство, всегда подтверждает основной владелец.

Подтверждение должно показывать продавца, состав заказа, итоговую сумму, валюту, адрес или получателя и существенные условия. Сохраненная карта, платежный токен или банковские credentials не передаются модели. Члены семьи могут подготовить покупку, но подтвердить платеж может только владелец.

### 12.8. Workspace

Каждый член семьи получает отдельный постоянный sandbox workspace. Семейные файлы хранятся в отдельном семейном workspace, а внешняя группа получает собственный изолированный workspace согласно своей политике.

Правила изоляции:

- личный workspace доступен только соответствующему пользователю;
- семейный workspace доступен подтвержденным членам семьи с учетом роли;
- workspace внешней группы не имеет доступа к личным или семейным файлам;
- изолированные файловые операции поиска, чтения и записи доступны во всех workspace без отдельного прикладного каталога;
- перенос файла между областями является явным действием и показывает область назначения;
- физический workspace является источником истины, а необязательные поисковые данные не блокируют файловые операции;
- удаление workspace или файла удаляет связанные производные данные и журналы операций по retention-политике;
- sandbox shell наследует ту же область workspace и не может перейти в соседнюю область.

### 12.9. Subagents

Основной агент может автоматически делегировать сложные задачи subagents без отдельного подтверждения, если само делегирование не создает внешний побочный эффект.

Для первой версии:

- одновременно запускается не более `3` subagents на один родительский ход;
- максимальная глубина делегирования равна `1`;
- subagent наследует область памяти, пользователя, роль, allowlist инструментов и ограничения родительской сессии;
- subagent не может повысить собственные права или получить другой workspace;
- использование subagent учитывается в бюджете инициатора;
- внешний побочный эффект subagent проходит ту же проверку и подтверждение, что и прямой вызов основного агента;
- основной агент сообщает пользователю итог делегирования без показа внутреннего reasoning.

### 12.10. Gmail

После Google Calendar подключается Gmail. Каждый участник может подключить несколько личных или рабочих Google-аккаунтов через OAuth-ссылку в личном Telegram-чате.

Агент может искать и читать письма, готовить сводки и создавать проекты ответов. Отправка письма выполняется только после подтверждения владельца соответствующего Gmail-аккаунта. Перед подтверждением показываются аккаунт отправителя, получатели, тема, текст и вложения.

Проактивные уведомления о важных письмах работают только по персональным правилам пользователя: отправители, адреса, темы или категории. Модель не включает уведомления о письмах по собственной оценке без такого правила.

## 13. Подтверждение действий

Human-in-the-loop Eve должен использовать Telegram inline-кнопки или явный ответ пользователя.

Подтверждение обязательно для действий, которые:

- отправляют сообщения, письма или публикации от имени пользователя;
- создают, изменяют или удаляют внешние данные;
- удаляют память;
- совершают покупки, платежи или иные финансовые операции;
- раскрывают чувствительную информацию;
- имеют юридические, медицинские или иные значимые последствия.

Подтверждение дает право только на конкретное действие с показанными параметрами. Инициатор группового запроса не может подтвердить действие от имени другого пользователя. Для действий с общей семейной собственностью политика может дополнительно требовать подтверждение владельца.

Каждое внешнее действие должно использовать идемпотентный ключ, когда сервис это допускает. Это защищает от повторного эффекта при возобновлении прерванного шага Eve.

## 14. Напоминания и проактивные сообщения

Агент должен поддерживать:

- личные одноразовые напоминания;
- личные повторяющиеся напоминания;
- семейные напоминания в зарегистрированной группе;
- отправку в конкретную тему группы;
- просмотр, изменение, приостановку и удаление напоминаний.

Перед созданием напоминания агент обязан определить часовой пояс, время и канал назначения. Неоднозначные значения нельзя заменять догадками.

Каждый участник задает собственный часовой пояс и тихие часы. Обычные уведомления, попадающие в тихие часы, откладываются до окончания интервала. Срочные уведомления могут обходить тихие часы только если пользователь заранее разрешил это для соответствующего типа событий.

Агент может проактивно сообщать о приближающемся событии, конфликте календаря, пропущенной задаче или другой полезной ситуации на основании разрешенных данных. Проактивность дает право предложить действие, но не отменяет правила подтверждения и не разрешает самостоятельно изменять внешние данные.

Семейное напоминание могут изменить, приостановить или удалить его автор и владелец семьи. Остальные участники могут просмотреть напоминание и предложить изменение.

Динамические расписания хранятся в прикладной БД. Диспетчер должен атомарно захватывать наступившие задания, использовать аренду и учитывать доставку как минимум один раз. Побочные эффекты должны быть идемпотентными.

Если срок напоминания прошел во время остановки агента, оно отправляется после запуска с явной отметкой о задержке. Одноразовое напоминание после успешной доставки закрывается. Повторяющееся напоминание не воспроизводит все пропущенные интервалы и продолжает работу со следующего будущего срока.

### 14.1. Семейные задачи

Семейные задачи и поручения хранятся в собственной PostgreSQL-базе агента без обязательной внешней интеграции.

Любой член семьи может создать личную задачу или назначить семейную задачу другому участнику. Закрыть семейную задачу могут исполнитель, автор и владелец. Если задача остается просроченной сутки, агент публикует уведомление в семейной группе.

### 14.2. Персональная утренняя сводка

Каждый участник самостоятельно включает сводку и выбирает время отправки с учетом своего часового пояса. Доступные блоки: календарь, задачи, важные письма и семейные события. Невключенный блок не собирается и не отправляется.

## 15. Безопасность и приватность

### 15.1. Входящий Telegram webhook

- Проверять `X-Telegram-Bot-Api-Secret-Token` до обработки тела как доверенного события.
- Использовать постоянное по времени сравнение секретов.
- Получать Telegram user ID и chat ID только из проверенного события.
- Не считать наличие пользователя в группе достаточным основанием для доступа к семейной области.
- Отклонять неизвестные группы и неподтвержденных пользователей до запуска модели.

### 15.2. Секреты

- Секреты хранятся только в менеджере секретов или переменных окружения, когда значение действительно секретное или зависит от окружения.
- Секреты не попадают в исходный код, логи, память, prompt или sandbox.
- Инструменты с секретами исполняются в доверенном runtime Eve.

### 15.3. Sandbox

- Sandbox не получает `process.env` и секреты.
- Сетевой доступ sandbox ограничивается явной политикой.
- Привилегированные запросы выполняются через инструменты или подключения.

### 15.4. Изоляция данных

- Идентификатор семейной области обязателен во всех операциях чтения и записи.
- Ограничения пользователя, семьи и группы применяются на уровне запроса к БД.
- Внешние идентификаторы не принимаются от модели как основание авторизации.
- Проверки доступа покрываются интеграционными тестами на межпользовательские и межсемейные утечки.

## 16. Концептуальная модель данных

Окончательная схема создается отдельной технической спецификацией и миграциями. Минимально необходимы сущности:

| Сущность | Назначение |
| --- | --- |
| `families` | Семейные области. |
| `users` | Внутренние пользователи. |
| `telegram_identities` | Связь проверенного Telegram user ID с пользователем. |
| `family_memberships` | Членство и роль пользователя в семье. |
| `telegram_groups` | Зарегистрированные группы, их тип, владелец, область памяти и политика инструментов. |
| `invitations` | Одноразовые приглашения и их состояние. |
| `memory_items` | Долговременные записи памяти и область доступа. |
| `scheduled_tasks` | Личные и семейные расписания. |
| `integration_accounts` | Метаданные подключений без хранения токенов в открытом виде. |
| `web_credentials` | Метаданные credential broker без доступных модели секретов. |
| `workspaces` | Личные, семейные и групповые области файлов. |
| `audit_events` | События безопасности и внешние действия. |

Токены OAuth должны храниться в предназначенном для этого защищенном хранилище или сервисе, а не как обычные поля `integration_accounts`.

## 17. Аудит и наблюдаемость

Журнал должен фиксировать:

- вход и отказ в доступе;
- приглашение, подтверждение и удаление участника;
- регистрацию и удаление группы;
- создание, изменение и удаление памяти;
- запрос, подтверждение, отклонение и результат внешнего действия;
- создание и запуск расписания;
- ошибки модели, инструмента, подключения и доставки.

Владелец может просматривать административные события и внешние действия семейной области. Содержание личных разговоров и личной памяти других участников не должно включаться в административный журнал по умолчанию.

Для технической наблюдаемости требуется структурированное логирование и OpenTelemetry. Пользовательские ошибки должны быть на русском языке, не содержать внутренних деталей и включать стабильный код.

Примеры обязательных кодов:

| Код | Пользовательское сообщение |
| --- | --- |
| `AGENT_ACCESS_DENIED` | «У вас нет доступа к этому семейному агенту. Попросите владельца отправить приглашение». |
| `AGENT_GROUP_NOT_REGISTERED` | «Эта группа не подключена к семейному агенту. Владелец может добавить ее в настройках». |
| `AGENT_APPROVAL_REQUIRED` | «Для выполнения этого действия требуется подтверждение». |
| `AGENT_INTEGRATION_AUTH_REQUIRED` | «Для выполнения действия подключите нужный сервис». |
| `AGENT_MEMORY_SCOPE_DENIED` | «Эта информация недоступна в текущем чате». |

Внутренние логи могут содержать технические причины, correlation ID и исходные коды провайдеров, но не секреты.

Технический аудит хранится `90` дней с момента события и затем автоматически удаляется. Исключение допускается только для события, вручную отмеченного владельцем как относящееся к незавершенному инциденту; после закрытия инцидента снова применяется обычный срок хранения.

## 18. Нефункциональные требования

- Runtime: Node.js 24 или версия, обязательная для закрепленного релиза Eve.
- Все обязательные настройки валидируются при старте; отсутствующая конфигурация приводит к остановке с понятной ошибкой.
- Долговечные данные хранятся на постоянном носителе и резервируются.
- Должна существовать простая проверенная команда восстановления из резервной копии.
- Миграции БД должны быть повторяемыми и выполняться контролируемо.
- Система должна иметь лимиты расходов на модель, размеры вложений, объем памяти и частоту запросов.
- Политики лимитов задаются именованными константами после продуктового согласования, а не неявными значениями.
- Критические пути покрываются unit- и integration-тестами; Telegram webhook, память, авторизация и подтверждения проверяются с моками внешних API.
- Производственные внешние действия должны быть идемпотентными или защищенными от повторного выполнения.

### 18.1. Бюджеты и квоты

При первом запуске владелец сам задает месячный семейный бюджет и лимиты отдельных пользователей; предустановленной денежной суммы нет. Бюджет учитывает переменные расходы моделей, embeddings, speech-to-text, внешних API и sandbox, когда провайдер предоставляет стоимость или измеримый usage.

Обязательное поведение:

- при достижении `70%` семейного или пользовательского бюджета отправляется первое предупреждение;
- при достижении `90%` отправляется критическое предупреждение владельцу и соответствующему пользователю;
- при достижении `100%` новые платные операции в затронутой области блокируются;
- бесплатные административные команды управления бюджетом, памятью и подключениями остаются доступны;
- auto-fallback не выполняется на маршрут, который нарушит действующий бюджет;
- владелец может повысить бюджет только явным подтвержденным действием;
- участник не может повысить собственный лимит;
- usage подписочных моделей учитывается отдельно от переменных API-расходов, даже если непосредственная цена отдельного запроса равна нулю;
- каждый пользователь может посмотреть собственное потребление, владелец видит агрегированное семейное потребление и техническую разбивку без текста личных запросов.

Денежную сумму владелец выбирает при первоначальной настройке и позднее может изменить подтвержденным действием. Технические rate limits должны защищать личный экземпляр от случайного зацикливания и спама, а не рассчитываться на высокую нагрузку.

## 19. Развертывание

Выбрано простое self-hosted развертывание личного агента через Docker Compose для локальной разработки и рабочего экземпляра.

### 19.1. Сервисы

Обязательные сервисы:

- Eve agent runtime;
- PostgreSQL с необходимыми расширениями;
- edge reverse proxy;
- model proxy для подписочных провайдеров;
- локальный multilingual embedding worker;
- локальный сервис безопасного разбора документов;
- выбранный sandbox backend;
- постоянный workflow storage или совместимый Workflow world.

### 19.2. Требования

- Рабочий образ агента собирается из канонического состояния репозитория.
- Reverse proxy передает маршруты `/eve/` и `/.well-known/workflow/` без нарушения streaming и workflow callbacks.
- Долговечные данные, workflow state и sandbox-данные с требуемым сроком жизни размещаются на именованных постоянных томах или во внешних хранилищах.
- PostgreSQL, runtime и reverse proxy недоступны из публичной сети напрямую, кроме маршрутов, явно опубликованных через edge proxy.
- Локальная разработка и рабочий экземпляр используют отдельные секреты, БД, Telegram bots и внешние OAuth clients.
- Миграции выполняются внутри контейнера агента контролируемой командой миграции.
- Обновление контейнера не должно уничтожать активные сессии, расписания или память.

### 19.3. Workflow storage и восстановление

Рабочий экземпляр запускается на одном сервере и использует локальный Workflow world Eve с постоянным томом `.workflow-data`. Высокая доступность, несколько runtime-инстансов и отдельные SLO не требуются.

Сервер считается CPU-only: наличие NVIDIA GPU не требуется и не должно быть условием запуска. Тяжелые языковые, vision- и speech-модели вызываются через model proxy/API; локально выполняются PostgreSQL, multilingual E5 embeddings и безопасный структурный разбор документов.

Обязательные требования:

- одновременно запускается только один рабочий runtime, владеющий локальным Workflow world;
- `.workflow-data` и PostgreSQL хранятся на постоянных томах;
- автоматическая резервная копия включает PostgreSQL и `.workflow-data`;
- восстановление выполняется одной документированной операторской командой;
- после восстановления проверяются целостность памяти, расписаний и возможность продолжить тестовую сессию;
- во время перезапуска или восстановления агент может быть временно недоступен.

### 19.4. Модельная подсистема

Модели подключаются через единый AI SDK provider registry. Прикладной код и агент используют логические идентификаторы моделей, а registry разрешает их в конкретный `LanguageModel`.

Обязательные маршруты:

- `codex-subscription`: основной GPT-5.5 через авторизованную подписку Codex и custom CLI Proxy API;
- `subscription-proxy`: подписочные GLM, Kimi, MiniMax и OpenCode Go через custom CLI Proxy API;
- `vercel-gateway`: модели, доступные через Vercel AI Gateway;
- `deepseek-api`: DeepSeek через отдельный API-ключ;
- дополнительные провайдеры через отдельные адаптеры без изменения доменной логики агента.

В качестве model proxy используется CLIProxyAPI. Он подключается как custom OpenAI-compatible AI SDK provider с настроенными `baseURL`, заголовком авторизации и стабильными model aliases. Если конкретная модель не полностью совместима с OpenAI API, для нее создается отдельный provider adapter, а несовместимость не маскируется общим интерфейсом.

Версия CLIProxyAPI закрепляется, а adapter покрывается контрактными тестами streaming, tool calling, reasoning metadata, ошибок авторизации и usage. Обновление proxy выполняется только после прохождения этих тестов.

Основным логическим маршрутом является GPT-5.5 через `codex-subscription`. Vercel AI Gateway остается доступным маршрутом, но не является единственной точкой доступа к моделям.

### 19.5. Учетные данные моделей

- Подписочные учетные данные находятся внутри model proxy и не передаются Eve runtime, модели, sandbox или памяти.
- Eve runtime получает только отдельный credential для обращения к model proxy.
- `AI_GATEWAY_API_KEY`, DeepSeek API key и credential model proxy хранятся как независимые секреты рабочего экземпляра.
- Каждый подписочный коннектор должен использовать разрешенный провайдером способ авторизации и не должен извлекать или сохранять пользовательские credentials в логах.
- Отсутствие обязательного credential для выбранного маршрута приводит к явной ошибке запуска или выбора модели.

### 19.6. Маршрутизация и отказоустойчивость моделей

Для каждого model alias должны быть заданы возможности: tool calling, streaming, reasoning, максимальный контекст, поддерживаемые типы входа и максимальный output.

Владелец задает упорядоченную цепочку разрешенного auto-fallback. Добавление модели в цепочку означает явное разрешение передавать ей данные при недоступности предыдущего маршрута.

Только владелец может выбирать основной model alias, изменять fallback-цепочку и назначать модели для семейных или внешних групп. Остальные участники используют утвержденную владельцем модельную политику и не могут переключать провайдера из диалога.

Начальная fallback-цепочка не задается системой. Владелец формирует ее из фактически подключенных моделей при первоначальной настройке; до этого пустая цепочка означает отсутствие auto-fallback.

Правила auto-fallback:

- переход допускается только на следующий заранее одобренный model alias;
- fallback запускается при недоступности маршрута, timeout или provider rate limit;
- ошибка обязательной конфигурации, отсутствующий credential, нарушение авторизации, несовместимость tool calling или отклонение запроса политикой провайдера не маскируются fallback;
- на один model call разрешен один переход без повторных скрытых ретраев;
- перед вызовом fallback проверяется совместимость модели с требуемыми tools, типами входа и контекстом;
- пользователь получает уведомление о фактической модели и причине переключения;
- событие фиксируется в аудите без содержимого prompt;
- если fallback-модель также недоступна, операция завершается диагностируемой ошибкой;
- пустая цепочка означает запрет auto-fallback.

### 19.7. Известные ограничения

- больше операционной ответственности;
- single-node runtime временно недоступен при перезапуске или отказе сервера;
- подписочные provider adapters зависят от совместимости и условий соответствующих провайдеров;
- сложный PostgreSQL Workflow world и кластеризация не входят в требования личного агента.

## 20. Критерии приемки первой вертикали

Система считается соответствующей первой вертикали, если:

1. Неизвестный Telegram-пользователь не может запустить модель или получить данные семьи.
2. Владелец может создать одноразовое приглашение и подтвердить участника.
3. Каждый участник может вести отдельный личный разговор с агентом.
4. Агент отвечает в зарегистрированной группе по команде, упоминанию или ответу на его сообщение.
5. Агент игнорирует неизвестную группу и не раскрывает в ней данные.
6. Группа `family_private` получает семейную память.
7. Сообщение пользователя без семейного membership не запускает модель в `family_private`.
8. Группы `external_private` и `external_public` получают только собственную групповую память.
9. Личная и семейная память недоступны во внешних группах.
10. Индивидуальная личная память не используется в `family_private`.
11. Любой принятый участник внешней группы может создать запись только в памяти этой группы с указанием авторства.
12. Автоматически созданная память сопровождается уведомлением и может быть немедленно удалена.
13. Семейную запись могут исправить или удалить только автор и владелец.
14. Память хранится до явного удаления и удаляется из полнотекстового и векторного индексов.
15. Гибридный поиск применяет область доступа до полнотекстового и векторного поиска.
16. Пользователь может создать, просмотреть, изменить, удалить и экспортировать свою память.
17. Внешняя группа может вызвать только инструмент из собственного allowlist.
18. Настройки внешней группы может изменять только владелец агента.
19. Удаление внешней группы удаляет ее память, workspace, файлы, сессии и индексы.
20. Полное удаление пользователя немедленно удаляет его личные данные из рабочего хранилища.
21. Выходящий из семьи взрослый получает экспорт, а его личные данные удаляются без передачи владельцу.
24. Google Calendar использует подключение текущего пользователя или явно выбранный семейный календарь.
25. Все взрослые могут изменять семейный календарь после подтверждения.
26. Изменение календарного события не выполняется без требуемого подтверждения.
27. Опасное действие не выполняется без требуемого подтверждения.
28. Чужая или просроченная кнопка подтверждения не выполняет действие.
29. Повторное выполнение шага не дублирует поддерживаемый внешний побочный эффект.
30. Основной alias разрешается в GPT-5.5 через `codex-subscription`.
31. Auto-fallback использует только заранее одобренную владельцем цепочку и уведомляет пользователя о переключении.
32. За один model call выполняется не более одного fallback-перехода без скрытых ретраев.
33. Только владелец может изменять основную модель и fallback-цепочку.
35. Подписочные credentials недоступны Eve runtime и sandbox.
36. Первый owner создается только действующим одноразовым bootstrap-кодом в личном чате.
37. Пользовательские и административные настройки доступны обычным языком в Telegram без обязательного каталога команд.
38. Повторное использование bootstrap-кода и bootstrap после создания owner отклоняются.
39. Личное напоминание доставляется в личный чат, а семейное — в зарегистрированную группу или тему.
40. Семейное напоминание могут изменить или удалить только автор и владелец.
41. Обычное уведомление соблюдает личные тихие часы получателя.
42. Пропущенное во время остановки напоминание доставляется после запуска с отметкой о задержке без воспроизведения всех пропущенных повторов.
43. Проактивное предложение не выполняет внешнее изменение без требуемого подтверждения.
44. Рестарт процесса не уничтожает долговечную сессию, память и расписания.
45. Ротация и compaction сессий соблюдают политику раздела 11.1.
46. Изображения, PDF и Office-документы до `20 МБ` на входе hosted Telegram Bot API проходят заданный безопасный pipeline и хранятся до явного удаления; workspace и исходящие документы поддерживают до `50 МБ`.
47. Embeddings строятся локально multilingual-моделью без передачи памяти внешнему API.
48. Изображения и визуальные страницы документов обрабатываются только выбранным владельцем vision-маршрутом.
49. Голос транскрибируется настроенным model proxy/API provider.
50. На голосовой запрос агент по умолчанию отвечает текстом.
51. Продолжительная задача показывает короткие статусы без внутреннего reasoning.
52. Credential broker не раскрывает пароли, cookies и 2FA secrets модели или sandbox.
53. Финальное действие покупки или платежа выполняется только после подтверждения основного владельца.
54. Личные, семейные и внешние workspace изолированы друг от друга.
55. Subagent наследует права родительской сессии и не повышает их.
56. На один родительский ход запускается не более `3` subagents с глубиной `1`.
57. Достижение `100%` бюджета блокирует новые платные операции, но не административное управление.
58. Только заранее подтвержденный резервный владелец может инициировать 48-часовое восстановление.
60. Ошибки имеют стабильные коды, понятные русские сообщения и техническую диагностику в логах.
61. Резервное копирование и восстановление данных подтверждены практической проверкой.
62. Восстановление владельца нельзя запустить без recovery code резервного владельца.
65. Внешний побочный эффект из внешней группы подтверждает только основной владелец.

## 21. Зафиксированные решения

- Использовать Eve как ядро исполнения агента.
- Использовать один центральный агент для разных каналов.
- Начать с Telegram.
- Выполнять все настройки первой версии в Telegram через обычный язык и inline-подтверждения.
- Поддерживать владельца и приглашенных членов семьи.
- Поддерживать личные чаты, закрытые семейные и изолированные внешние группы.
- В группе реагировать по команде, упоминанию или ответу на сообщение бота.
- Разделять личную, семейную, групповую и служебную память.
- Использовать семейную память в `family_private` и только изолированную групповую память во внешних группах.
- Не использовать индивидуальную личную память в `family_private`.
- Принимать обращения в `family_private` только от подтвержденных членов семьи.
- Разрешить участникам внешней группы создавать память только этой группы с указанием авторства.
- Разрешить менять настройки внешней группы только владельцу агента.
- Разрешить исправлять и удалять семейную память автору записи и владельцу.
- Немедленно удалять личные данные пользователя после подтвержденного запроса полного удаления.
- Предлагать взрослому экспорт перед выходом из семьи и не передавать его личные данные владельцу.
- Удалять память, файлы, workspace и сессии внешней группы вместе с группой.
- Не раскрывать владельцу личные разговоры и личную память других участников автоматически.
- Хранить долговременную память вне Eve session state.
- Использовать подтверждение для опасных действий.
- Проверять инициатора, полномочия и срок действия группового подтверждения.
- Поддерживать личные и семейные напоминания.
- Разрешить агенту проактивно предлагать полезные действия без обхода подтверждений.
- Настраивать часовой пояс и тихие часы отдельно для каждого участника.
- Разрешить автору и владельцу управлять семейным напоминанием.
- Доставлять пропущенное напоминание после запуска с отметкой о задержке.
- Использовать проверенную Telegram-идентичность и подтверждаемое приглашение.
- Проектировать подключения как личные или семейные.
- Разрешить членам семьи web, файлы, sandbox shell и subagents в пределах безопасностных ограничений.
- Подключать web-аккаунты через credential broker без передачи секретов модели.
- Разрешить агенту подготавливать покупки, но требовать подтверждение основного владельца для финансового действия.
- Подтверждать внешние побочные эффекты гостевых и рабочих групп только основным владельцем.
- Использовать отдельные постоянные workspace пользователя, семьи и внешней группы.
- Разрешить автоматическое делегирование не более чем `3` subagents с глубиной `1` и наследованием прав.
- Настраивать отдельный allowlist инструментов для каждой внешней группы.
- Автоматически сохранять допустимые устойчивые воспоминания, уведомлять о записи и давать возможность немедленного удаления.
- Ротировать сессии и хранить историю согласно разделу 11.1.
- Поддержать одного заранее подтвержденного резервного владельца для восстановления.
- Проверять вложения и управлять сроком жизни всех производных данных.
- Поддержать изображения, PDF, Office-документы и голосовые сообщения в первой версии.
- Ограничить входящий Telegram-файл `20 МБ`, workspace и исходящий документ `50 МБ`, и хранить оригинал до явного удаления.
- Строить embeddings локальной `intfloat/multilingual-e5-small` на CPU.
- Использовать выбранную vision-модель для изображений и визуальных страниц документов.
- Транскрибировать голос через model proxy/API.
- Не требовать GPU для рабочего сервера.
- Использовать теплый адаптивный стиль и краткие ответы по умолчанию.
- Отвечать на голосовые сообщения текстом, если пользователь не попросил голосовой ответ.
- Показывать короткие статусы продолжительных задач без внутреннего reasoning.
- Хранить долговременную память до явного удаления.
- Использовать гибридный поиск PostgreSQL full-text search и `pgvector`.
- Использовать месячный семейный бюджет и пользовательские лимиты с блокировкой платных операций при достижении `100%`.
- Использовать Google Calendar как первую внешнюю интеграцию.
- Подключать личный Google Calendar по OAuth-ссылке в личном чате, а семейный календарь выбирает владелец.
- Разрешить всем взрослым изменять семейный календарь после подтверждения.
- Подключить Gmail после Calendar: несколько аккаунтов, чтение, сводки и проекты ответов с подтверждаемой отправкой.
- Уведомлять о важных письмах только по персональным правилам пользователя.
- Хранить семейные задачи в собственной PostgreSQL-базе.
- Разрешить создавать и назначать задачи всем членам семьи; закрывать задачу исполнителю, автору и владельцу.
- Публиковать в семейной группе задачу, просроченную на сутки.
- Поддержать персональную утреннюю сводку из календаря, задач, важных писем и семейных событий.
- Развертывать локальный и рабочий экземпляры через Docker Compose.
- Использовать один runtime и постоянный локальный Workflow world без требований высокой доступности.
- Использовать AI SDK provider registry для всех модельных маршрутов.
- Использовать CLIProxyAPI как закрепленный model proxy с контрактными тестами.
- Использовать GPT-5.5 через Codex-подписку и CLI Proxy API как основной модельный маршрут.
- Поддержать подписочные GLM, Kimi, MiniMax и OpenCode Go через custom provider.
- Поддержать DeepSeek через API-ключ и Vercel AI Gateway как отдельные маршруты.
- Выполнять auto-fallback только по заранее одобренной владельцем цепочке с уведомлением пользователя.
- Разрешить изменять основную модель и fallback-цепочку только владельцу.
- Не создавать fallback-цепочку по умолчанию; владелец настраивает ее при первом запуске.
- Создавать первого владельца одноразовым Telegram bootstrap-кодом.
- Использовать 48-часовую задержку восстановления владельца.
- Требовать recovery code резервного владельца для запуска восстановления.
- Хранить технический аудит `90` дней.
- Хранить активные skills только как проверенные файлы в `agent/skills`, без содержимого skills в БД.
- Создавать runtime-черновики skills только как неизменяемые SHA-256-ревизии в sandbox.
- Активировать новый skill только через ручную проверку точных байтов, create-only promotion, полный тестовый прогон и `eve build`.

## 22. Открытые решения

Перед реализацией необходимо согласовать:

1. Точная версия CLIProxyAPI, model IDs и protocol capabilities каждого подписочного маршрута.
2. Точный порядок отмены 48-часового восстановления резервным владельцем.
3. OAuth scopes, выбор календарей и остальные семейные правила Google Calendar.
4. Технические rate limits, количество файлов в сообщении и ограничения объема памяти.
5. Точные speech-to-text и vision model aliases. E5 model revision закреплена в `docs/memory.md`.
